💡 Ключевая идея

Один движок — много бизнесов. Каждый клиент (тенант) получает своего бота, который выглядит и работает как уникальный продукт, но под капотом — единая платформа.

🔄 Как это работает

Представь WordPress: один движок, тысячи сайтов с разными темами и плагинами. Так же и здесь:

  • Платформа — FSM engine, LLM client, webhook router, quality system, analytics
  • Тенант-конфиг — system prompt, правила бизнеса, интенты, few-shot примеры, стиль общения
  • Плагины — адаптеры каналов (VK, Telegram, WhatsApp), интеграции с CRM (DIKIDI, Shiftledger)

🔑 Главный принцип

Если написал код — это platform, работает у всех. Если написал текст/JSON — это tenant config, специфичен для бизнеса. Код никогда не знает, что он работает для танцевальной студии или детского центра. Он работает с абстракциями: intent, entity, rule, response.

🔧 Ключевой код — TenantService

// src/platform/tenant/tenant.service.ts
// Это сердце мульти-тенантности

class TenantService {
  private configs = new Map<string, TenantConfig>();

  // Загрузка при старте
  async loadAll() {
    const dirs = fs.readdirSync('src/tenants');
    for (const dir of dirs) {
      const config = await this.loadTenantConfig(dir);
      this.configs.set(config.id, config);
    }
  }

  // Маршрутизация webhook → tenant
  resolveByWebhook(path: string): TenantConfig {
    // /webhook/nexa/telegram → tenantId = "nexa"
    // /webhook/pro_deti/vk   → tenantId = "pro_deti"
    const tenantId = path.split('/')[2];
    return this.configs.get(tenantId);
  }

  // Получение конфига
  getConfig(tenantId: string): TenantConfig {
    return this.configs.get(tenantId);
  }
}

🔧 Engine — не знает о бизнесе

// src/platform/engine/engine.service.ts

async process(tenantId: string, message: UnifiedMessage): Promise<BotResponse> {
  const config = this.tenantService.getConfig(tenantId);

  // 1. Классификация — промпт из конфига тенанта
  const intent = await this.classifier.classify(
    message.text,
    config.intentClassifierPrompt,  // тенант-специфичный
    config.intents                   // список интентов тенанта
  );

  // 2. FSM — конфиг из тенанта
  const action = this.fsm.transition(
    conversation.state,
    intent,
    config.fsmConfig                 // переходы и хендлеры тенанта
  );

  // 3. Генерация — system prompt из тенанта
  if (action.type === 'auto_response') {
    return config.templates[intent];  // шаблон тенанта
  }

  return this.generator.generate(
    config.systemPrompt,             // промпт тенанта
    config.rules,                    // правила бизнеса тенанта
    config.fewShotExamples[intent],  // примеры тенанта
    message, context
  );
}

🛤️ Как перейти от NEXA к платформе

Ты продолжаешь работать над NEXA — но каждая доработка сразу делается «платформенно».

Вынеси NEXA-специфичное в конфиг

Создай папку src/tenants/nexa/. Перенеси туда: system prompt, knowledge base, FSM config, few-shot примеры. Код в src/platform/ начинает читать конфиг вместо хардкода. NEXA продолжает работать как раньше.

Добавь TenantService

Простой сервис: загружает конфиг по tenantId, резолвит webhook → tenant. Один тенант (nexa) — всё работает. Но архитектура уже готова для второго.

Добавь tenant_id в Prisma

В модели Customer, Conversation, Message — добавь tenantId String. Миграция для существующих данных: UPDATE SET tenantId = 'nexa'. Все запросы через where: { tenantId }.

Добавь VK adapter

Telegram adapter уже есть. Напиши VK adapter по тому же интерфейсу. Оба реализуют ChannelAdapter. Это platform-код, доступен всем тенантам.

Создай тенант «Про Дети»

Скопируй src/tenants/nexa/ → src/tenants/pro_deti/. Замени конфиги на подготовленные (system_prompt, fsm_config, few_shot — уже готовы!). Зарегистрируй webhook: /webhook/pro_deti/vk. Всё.

Profit

Любая доработка в src/platform/ — улучшает обоих ботов. Новый клиент = новая папка в src/tenants/. Подключение занимает часы, не недели.

✅ Что НЕ нужно делать

  • Не нужно переписывать NEXA — эволюционный рефакторинг
  • Не нужно microservices — один NestJS-сервис с tenant routing
  • Не нужно отдельные БД per tenant — одна БД с tenant_id
  • Не нужно отдельные деплои — один Render сервис
