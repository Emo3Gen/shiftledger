# Аудит бизнес-логики ShiftLedger

Сравнение текущей реализации с реальным бизнес-процессом детского центра (по словам директора Семёна).

---

## Что ПРАВИЛЬНО в текущей реализации

1. **NL-парсер свободного текста** (`factsParserV0.js`)
   - Распознаёт "могу пн утро", "не могу ср вечер", "свободна пт день" и т.д.
   - Короткие формы ("пн 10-13", "вт вечер") — как в реальном чате.
   - Именно так сотрудники пишут в чат: свободным текстом, без кнопок и форм.

2. **Facts pipeline (events → facts)**
   - Архитектура `public.events` → `public.facts` корректна.
   - Идемпотентность через `fact_hash`, версионирование парсера (`parser_version`).
   - Позволяет пересчитывать факты без потери данных.

3. **Schedule engine с авто-назначением** (`scheduleEngineV0.js`)
   - 3-фазный алгоритм: минимальные часы → равномерное распределение → seniors как резерв.
   - Использует `UserDirectory.getMinHours()` для учёта договорённостей.
   - Генерирует `reason` для каждого назначения — полезно для прозрачности.

4. **WeekHoursTemplate** — исторические часы по слотам (утро/вечер × день недели).
   - Соответствует реальности: "утро" = первые две утренние группы, обычно 5-5.5 часов.

5. **UserDirectory** — единый справочник сотрудников с ролями, ставками, minHours.
   - Синхронизация из БД при старте сервера.

6. **Timesheet** (`timesheetV0.js`) — расчёт ЗП по итоговому графику.
   - "План = факт" по умолчанию (как директор и хотел: считает по итоговому графику).

7. **Telegram-бот** — читает сообщения, парсит факты, показывает график.
   - Русский интерфейс, русские команды.

## Что ЛИШНЕЕ / мешает естественной работе

### 1. State machine с 6 состояниями (weekStateV0.js)

**Текущее:** DRAFT → COLLECTING → PROPOSED → CONFIRMING → LOCKED → EMERGENCY

**Реальность:** Бот спросил → сотрудники ответили → график готов → замены идут сами.

Лишние состояния:
- **PROPOSED** — нет этапа "предложения". Бот составил — отправил.
- **CONFIRMING** — сотрудники НЕ утверждают график. Они видят итог.
- **LOCKED** — нет формальной "блокировки". График живой, замены идут постоянно.
- **EMERGENCY** — это не отдельное состояние, а обычная ситуация "надо найти замену".

**Нужно:** Максимум 3 состояния: COLLECTING → ACTIVE → CLOSED.

### 2. Кнопки подтверждения и блокировки

**Текущее:** Эндпоинты `confirm-user`, `WEEK_CONFIRM`, `WEEK_LOCK`, `WEEK_PROPOSE`, `CONFIRM_SCHEDULE`.

**Реальность:** Директор НЕ утверждает график, НЕ нажимает кнопки. Сотрудники НЕ подтверждают. Бот составил — показал — всё.

Лишние fact types:
- `WEEK_PROPOSE` — нет этапа предложения
- `WEEK_CONFIRM` — нет этапа подтверждения
- `SCHEDULE_CONFIRMED` — сотрудники не подтверждают
- `OWNER_APPROVAL` — директор не утверждает отдельные смены
- `SHIFT_MARKED_PROBLEM` — нет ручной маркировки проблем

### 3. SHIFT_SWAP_REQUEST с target_user_id

**Текущее:** Формальный swap request: "SWAP mon 10-13 WITH isa".

**Реальность:** Замены происходят через живой чат: "Девочки, я не могу в чт утро, кто сможет?" → "Я смогу!". Никаких формальных запросов.

### 4. DSL-команды

**Текущее:** AVAIL, CANT, SWAP, GAP, LOCK, ASSIGN, WORKED, NO_SHOW, CONFIRM_SCHEDULE, REPORT_PROBLEM, MARK_SHIFT_PROBLEM, CONFIRM_SHIFT_FACT, OPEN_WEEK, PROPOSE, DECLARE_GAP, APPROVE_OVERTIME, CONFIRM_TIMESHEET — 17 команд.

**Реальность:** Сотрудники пишут СВОБОДНЫМ ТЕКСТОМ. DSL полезен только для внутренней отладки, но засоряет парсер и тесты.

### 5. Slot status machine (EMPTY → PENDING → CONFIRMED)

**Текущее:** В `scheduleEngineV0.js` каждый слот имеет статус: EMPTY, PENDING, CONFIRMED. Логика определения CONFIRMED vs PENDING сложная (сравнение timestamps подтверждений и назначений).

**Реальность:** Слот либо заполнен, либо нет. Нет этапа подтверждения.

### 6. Карина minHours = 0 в UserDirectory

**Текущее:** `u4/karina` имеет `minHours: 0` и `role: "senior"`.

**Реальность:** Карина ВСЕГДА работает больше 20 часов в неделю (предварительная договорённость). Она не "senior в резерве", а основной сотрудник с максимальной нагрузкой.

## Что ОТСУТСТВУЕТ

### 1. Бот сам инициирует сбор доступности
**Нужно:** Раз в неделю (по расписанию) бот пишет в чат:
"Девочки, собираю доступность на неделю 10-16 марта. Пожалуйста, напишите, когда можете работать. Срок — до вторника 18:00."

### 2. Бот задаёт уточняющие вопросы
**Нужно:** Если парсер не уверен (confidence < 0.7 или неполные данные), бот спрашивает:
"Дарина, правильно ли я понял: ср не можешь, вт только утро?"

### 3. Распознавание замен из переписки
**Нужно:** Бот читает чат и распознаёт цепочку замены:
1. "Я не могу в чт утро" → SHIFT_UNAVAILABILITY + триггер поиска замены
2. "Я смогу" / "Я выйду" (в ответ) → SHIFT_REPLACEMENT
3. Новый fact_type: `SHIFT_REPLACEMENT { date, dow, from, to, replaced_user_id, replacement_user_id }`

### 4. Предварительные договорённости
**Нужно:** Карина: minHours >= 20. Система должна СНАЧАЛА заполнить её часы, потом распределять остальных.
Текущий `scheduleEngineV0` уже поддерживает `minHoursRequirements`, но для Карины стоит 0.

### 5. Уведомление о свободных сотрудниках при отсутствии замены
**Нужно:** Если никто не откликнулся на запрос замены за N часов:
"Смена Чт утро не закрыта. Свободны: Карина, Дарина."

### 6. Автоматическая отправка графика в чат
**Нужно:** После составления графика бот отправляет его в Telegram-чат (не только показывает по команде /schedule).

### 7. Напоминания не ответившим
**Нужно:** Если сотрудник не ответил на запрос доступности через 24ч — бот напоминает.
Если до дедлайна не ответил — бот составляет график без него и уведомляет директора.

### 8. Автоматическое закрытие недели
**Нужно:** Переход ACTIVE → CLOSED когда неделя прошла (по дате). Данные фиксируются для расчёта ЗП.

### 9. Контекст ответа (reply-to)
**Нужно:** Бот должен понимать, что "Я смогу" — это ответ на конкретный запрос замены, а не общая доступность. Требует отслеживания reply-to в Telegram.
