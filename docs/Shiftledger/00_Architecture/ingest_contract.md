## Ingest Contract — входной JSON для `/ingest`

Этот контракт описывает **логический** формат события, которое попадает в ingest‑слой и далее в `public.events`.

### Входной JSON

Минимальный универсальный контракт:

- **Обязательные поля**
  - `tenant_id: string` — идентификатор арендатора (детский центр / сеть).  
  - `channel: string` — канал / интеграция (`telegram`, `mobile`, `web`, `emu`, ...).  
  - `chat_id: string` — логический идентификатор диалога/чата внутри tenant.  
  - `user_id: string` — логический идентификатор пользователя внутри tenant.  
  - `message_id: string` — идентификатор сообщения в рамках канала.  
  - `ts: string (ISO 8601)` — время, когда сообщение было отправлено пользователем.  
  - `text: string` — текстовое содержимое сообщения.

- **Опциональные поля**
  - `meta: object` — любые дополнительные данные, специфичные для канала/клиента.  
    Примеры:
    - `meta.role` — роль отправителя (`staff`, `admin`, `parent`, ...).  
    - `meta.telegram` — оригинальный update от Telegram (обрезанный / нормализованный).  
    - `meta.client` — версия мобильного/веб‑клиента.

### Гарантии ingest‑слоя

- Если запрос принят и прошёл валидацию:
  - Генерируется или принимается снаружи `trace_id` (из заголовка `x-trace-id`, если есть).  
  - Событие будет сохранено в `public.events` **ровно один раз** (с точки зрения бизнес‑семантики) или будет отвергнуто целиком.
  - В таблице `public.events` хранятся:
    - нормализованные поля из контракта;  
    - `source`/`channel` как часть `meta` или отдельным полем;  
    - `received_at` — когда событие дошло до backend.

- Что **не** гарантируется:
  - Порядок событий (`ts` и `received_at` могут отличаться, возможны задержки доставки).  
  - Мгновенный парсинг в facts (парсинг может быть отложенным).  
  - Единственность `message_id` глобально — она задаётся на уровне tenant+channel.

### Связь с текущей реализацией

- Текущий endpoint `/ingest` в `backend/server.js` реализует минимальное подмножество контракта:
  - обязательные: `chat_id`, `user_id`;  
  - опциональные: `source`, `text`, `meta.role`, `meta.*`;  
  - `tenant_id`, `channel`, `message_id`, `ts` могут быть добавлены позже как новые поля `meta`.
- Внутренняя схема `public.events` уже готова к расширению `meta`, поэтому переход к полному контракту возможен без миграции схемы.
