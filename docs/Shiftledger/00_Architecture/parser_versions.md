## Parser Versions — стратегия версионирования

Парсер превращает raw‑события (`public.events`) в facts (`public.facts`). Изменения в правилах парсинга должны быть управляемыми и отслеживаемыми.

### Идентификатор версии

- Версия парсера хранится в поле `parser_version` таблицы `public.facts`.  
- Формат: строка `vN`, где `N` — целое число, начиная с `0`:
  - `v0` — текущий детерминированный парсер (`factsParserV0.js`).  
  - `v1`, `v2`, ... — будущие итерации.

### Что считается breaking change

Изменения, требующие **новой** версии парсера:

- Изменение структуры `fact_payload` для существующего `fact_type`.  
- Изменение семантики поля (например, `availability: "can" | "cannot"` → `"yes" | "no"`).  
- Переименование `fact_type`.  
- Любые правки, которые могут изменить вывод парсера для уже сохранённых событий таким образом, что старые и новые факты **нельзя** сравнивать “в лоб”.

Изменения, которые можно делать в рамках существующей версии:

- Добавление новых `fact_type`, которые не конфликтуют со старыми.  
- Уточнение правил, не меняющее структуру `fact_payload` и не нарушающее инварианты.  
- Тюнинг `confidence`, если он используется только как soft‑сигнал.

### Правила миграции

- При выпуске `vN+1`:
  - Текущий runtime‑ингест начинает писать факты с новым `parser_version`.  
  - Старые факты с `parser_version = vN` **не удаляются**, используются как историческая трассировка.

- Для пересчёта (reparse):
  - Batch‑процессы могут:
    - либо добавлять новые факты с `parser_version = vN+1`,  
    - либо помечать старые факты статусом (`status = 'superseded'`/`'replaced'`).  
  - Важно: idempotency обеспечивается через `fact_hash`, который включает `parser_version`.

### Инварианты

- Один raw‑event (`event_id`) может иметь несколько наборов facts для разных `parser_version`.  
- В любом отчёте/фронте нужно явно указывать, какие `parser_version` считаются “источником правды” для расчётов.
