## Decision: Facts Pipeline v1

### Контекст

- В проекте Shiftledger нужен стабильный ingestion‑слой для чат‑сообщений (Telegram, мобильное приложение и др.).  
- Требуется детерминированно извлекать структурированные факты (смены, уборки, доп.занятия) для расчёта зарплат и операционного контроля.  
- Уже есть Supabase‑схема с таблицей `public.events` и первый парсер `factsParserV0.js`.

### Решение

- Фиксируем **events** как single source of truth: все входящие сообщения пишутся в `public.events` через endpoint `/ingest`.  
- Поверх событий строим слой **facts** в таблице `public.facts`, куда пишутся производные факты с `parser_version` и `fact_hash`.  
- Парсер версионируем (начиная с `v0`), любые breaking‑изменения ведут к новой версии.  
- Идемпотентность достигается через `fact_hash` и `upsert` по нему.  
- Для пересчёта (reparse) используем отдельные ручки/процессы, которые повторно читают `public.events` и пишут facts с новой версией.

### Последствия

- Можно безопасно развивать правила парсинга без потери истории и без дублей facts.  
- Любой баг в парсере можно исправить через reparse, не модифицируя raw‑events.  
- Появляется чёткий контракт для downstream‑систем (они читают только `public.facts` нужной версии).  
- Усложняется схема: нужно следить за статусами facts и версиями парсера.

### Next steps

- Добавить batch‑процесс для массового reparse по периоду/tenant.  
- Подключить “Chat Simulator” для интерактивной отладки ingest+parse+facts.  
- Зафиксировать правила версионирования парсера и требования к обратной совместимости (см. `parser_versions.md`).
