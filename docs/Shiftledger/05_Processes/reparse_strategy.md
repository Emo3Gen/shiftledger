## Reparse Strategy — пересчёт facts

Пересчёт нужен, когда:

- обновляется парсер (`parser_version` меняется с `vN` на `vN+1`);  
- меняются бизнес‑правила (ставки, типы смен), и нужно пересчитать derived‑данные;  
- исправляются баги в правилах распознавания.

### Сценарии пересчёта

- **On‑demand reparse одного события**
  - Endpoint `/parse/:eventId` уже реализован в `backend/server.js`.  
  - Загружает `event` из `public.events`, прогоняет через текущий парсер и пишет facts.  
  - Используется для точечной отладки и ручного восстановления.

- **Batch‑reparse по фильтру**
  - Отдельный процесс/скрипт (будет добавлен позже) выбирает нужные `event_id` из `public.events` (по периоду, tenant, каналу).  
  - Для каждого события вызывает парсер нужной версии и пишет facts с новым `parser_version`.

### Идемпотентность

- Для каждого факта вычисляется `fact_hash` (детерминированный sha256 по `event_id`, `fact_type`, `fact_payload`, `parser_version`).  
- Запись в `public.facts` делается через `upsert` по `fact_hash`, поэтому:
  - повторный запуск пересчёта не создаёт дубликаты;  
  - одинаковый факт (по содержимому) для того же события и версии всегда даёт одну и ту же строку.

### Что хранится в `facts`

- Каждый факт содержит:
  - ссылку на событие (`event_id`),  
  - тип (`fact_type`),  
  - строго структурированный `fact_payload`,  
  - `parser_version` и `status`,  
  - `trace_id`, `chat_id`, `user_id` для удобного дебага и агрегаций.

- `status` используется для жизненного цикла факта:
  - `parsed` — сгенерировано парсером и ещё не проверено человеком/процессом.  
  - `confirmed` — проверено и принято.  
  - `rejected` — признано ошибочным.  
  - `applied` — отражено в downstream‑системе (например, в сводке зарплат).

### Высокоуровневая стратегия

- Никогда не правим `public.events` “задним числом”, кроме откровенных технических ошибок.  
- Любой change‑log логики отражаем в версиях парсера и статусах facts.  
- Пересчёт facts всегда идемпотентен и может быть безопасно перезапущен.
